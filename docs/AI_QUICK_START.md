## 📐 開発の主軸：MDCS（ミセサポ演繹法分類体系）

### 論理構造（3段階）

1. **三段論法（前提の正確性の確保）**
   - 前提条件の正確性を確かめる
   - 分類方法間の一貫性を確保
   - **論理構造**: 要素AがグループBに属し、要素AがグループCに属するなら、グループBとグループCは同じ概念グループである（B ≡ C）

2. **演繹法（前提から結論への展開）**
   - 前提条件から結論を導く
   - 基盤システムから個別のシステムへと論理的に展開

3. **末広がりな思想（演繹的拡張）**
   - 基盤から始まって、徐々に広がっていく
   - 帰納法的収束ではなく、演繹的末広がり

### 基本原則

- **前提条件の確保**: どの分類方法を使っても、同じ要素が同じ概念グループに属する
- **演繹的展開**: 一般的な原理から個別のシステムへと論理的に展開
- **末広がりな拡張**: 小さな基盤から大きなシステムへと段階的に拡張

## 🏗️ システム概要

### アーキテクチャ

- **フロントエンド**: GitHub Pages（静的サイト）
- **バックエンド**: AWS Lambda + DynamoDB
- **認証**: AWS Cognito
- **リポジトリ**: GitHub（公開）

### システムの3層分類

#### 1. 顧客接点システム（Customer Interface）
- **目的**: 顧客との接点、新規顧客獲得、顧客情報管理
- **包含**: パブリック関連（コーポレートサイト）+ ユーザー管理（顧客情報）
- **特徴**: マーケティング的側面、SEO重要、デザイン重要

#### 2. 業務実行システム（Business Operations）
- **目的**: 現場での業務実行、業務プロセスの支援
- **包含**: 営業マンのツール + 清掃員のユーティリティツール
- **特徴**: 業務効率化、モバイル対応、即時性重要

#### 3. 管理統合システム（Management & Control）
- **目的**: 全業務の統合管理、意思決定支援、権限管理
- **包含**: 社内管理システム
- **特徴**: データ統合、分析、セキュリティ重要

### 主要なロール

1. **guest（ゲスト）**: 未認証ユーザー
2. **customer（顧客）**: 個人・法人の顧客
3. **staff（清掃員）**: 清掃作業を行うスタッフ
4. **sales/concierge（営業・コンシェルジュ）**: 営業担当者
5. **admin（管理者）**: システム管理者
6. **developer（開発者）**: 技術者・開発者
7. **master（マスター）**: 最上位権限

### 開発フェーズ（優先順位）

1. **Phase 1: 基盤システム**
   - 認証・認可、データベース、API基盤

2. **Phase 2: 業務実行システム（清掃員）**
   - 清掃員のユーティリティツール

3. **Phase 3: 業務実行システム（営業）**
   - 営業マンのツール

4. **Phase 4: 顧客接点システム**
   - パブリック関連 + ユーザー管理

5. **Phase 5: 管理統合システム**
   - 社内管理システム

**詳細**: [システム機能・ロール別カテゴライズ](./SYSTEM_CATEGORIZATION.md)

## 🔑 開発時の重要なポイント

### 1. 論理的一貫性の確保

- どの分類方法を使っても、同じ要素が同じ概念グループに属する
- 前提条件を確保してから、演繹的に展開する
- 帰納法的収束ではなく、演繹的末広がりを実現する

### 2. 「失敗しても良いシステム」の実現

- 失敗の影響範囲を限定する
- 失敗から回復を容易にする
- 段階的に構築することで、リスクを最小化する

### 3. 段階的な拡張

- 小さな基盤から大きなシステムへと段階的に拡張
- 各段階で論理的一貫性を保ちながら拡張
- 前提条件を確保してから、次の段階へ進む
