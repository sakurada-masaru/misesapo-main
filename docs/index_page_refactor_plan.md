## Indexページ整備方針

### 目的
- `src/` をソース・`public/` をビルド成果物とする既存運用を維持しつつ、Indexページを段階的に整理して保守性を高める。
- レイアウト/パーツ/データの責務を意識して、将来テンプレートエンジンやフレームワークへ移行しやすい構造にする。

### ディレクトリ方針（再確認）
| 役割 | ディレクトリ | 補足 |
| --- | --- | --- |
| ページソース | `src/pages/` | `index.html` を含む。原則ここで編集し、`public/` はビルド専用。 |
| 共通レイアウト | `src/layouts/` | 既存の `base.html` を活用／拡張する。 |
| パーツ（ヘッダー等） | `src/partials/` | ヘッダー/フッター/CTAなど共通化できる断片を切り出す。 |
| アセット | `src/assets/` | CSS/JS/画像。共通スタイルはここに置き、ページ固有スタイルはモジュール化を検討。 |

### Indexページ改善ステップ
1. **セクション棚卸し**: index内の主要セクション（ヒーロー、ニュース、CTA、紹介等）を洗い出し、役割と依存している CSS/JS を整理する（進行中のタスク）。
2. **スタイル分離**: 再利用しやすいスタイルは `src/assets/css/` へ移動。ページ固有のものもファイル単位で切り出し、ビルドで結合する。
3. **パーツ化**: 完全に共通化できる UI（例: 固定ボタン、SPメニュー、ニュースカード）を `src/partials/` に置き、`@include` で読み込めるよう準備。
4. **データ分離**: お知らせやカードリストなど静的データは `src/data/` に JSON として置き、将来的に `@json` 経由で差し込めるようにする。
5. **JS整理**: イベントリスナーやアニメーション制御をモジュール化し、`src/assets/js/index/` などにまとめる。DOM Ready で発火する初期化関数を統一。
6. **テスト/ビルド確認**: 各段階で `python3 scripts/build.py` → ローカルプレビューで差分を検証し、GitHub Upload Checklist を踏む。

### 当面のアクション
- Indexページの CSS/DOM をブロック単位で可視化し、優先的に整理すべき箇所を決める。
- 影響範囲が大きいセクションから順にスタイルを切り出し、HTML構造を読みやすくする。
- 変更内容と理由はこのドキュメントか各 PR の概要に追記して共有する。

### メモ
- `public/` 直編集はNG。常に `src/` 側を更新してビルドする。
- スプラッシュやモーダル等で `z-index` が高い要素が多いため、オーバーレイ順序をドキュメント化しておくとバグ診断が楽になる。
- 大規模編集前は `git status` を確認し、作業ブランチを分けることを推奨。
