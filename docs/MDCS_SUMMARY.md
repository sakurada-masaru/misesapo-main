## 🎯 MDCSの概要

**MDCS（ミセサポ演繹法分類体系）**は、システム開発と会社運営の核となる論理構造です。

**基本思想**: 「失敗しないシステムではなく、失敗しても良いシステム」

**MDCSの本質的な目的**:
- **失敗を恐れない開発環境の構築**: 厳格にするのは失敗しないためではなく、失敗を恐れないため
- **属人化の廃止**: 論理構造により、誰でも同じ判断ができる
- **相互補完的な開発環境**: 一人がみんなをカバーし、みんなが一人をカバーする状況を論理的に体現
- **層の厚い開発環境**: 複数の人が同じ領域を理解し、カバーできる環境を整える

**MDCSの本質**:
- **このシステムは一生完成することはない**: 常に可能性を秘めているから
- **完璧を目指すのではなく、常に改善し続ける**: 評価スコアが90でも問題ない、なぜならそれが正解だから
- **成長し続けるシステム**: 常に新しい可能性を秘め、進化し続ける

## 📐 論理構造（3段階）

### 1. 三段論法（前提の正確性の確保）

**役割**: 前提条件の正確性を確かめる

**MDCSでの三段論法の構造**:
- **大前提**: 要素Aが分類方法1のグループBに属する（A ∈ B）
- **小前提**: 要素Aが分類方法2のグループCに属する（A ∈ C）
- **結論**: したがって、グループBとグループCは同じ概念グループを表している（B ≡ C）

**システムとプログラムの関係性（三段論法の実践例）**:
- **A = システム（原理）**: システムは原理であり、システムだけでは運用できない
- **B = プログラム（原則）**: プログラムは原則であり、運用法則。法則だけではシステムは成り立たない
- **AはB**: システムとプログラムは同義であり、AはBが成立する
- **結論**: したがって、返される値として、僕が作るシステムはC（システムとプログラムの統合）にならなければならない

### 2. 演繹法（前提から結論への展開）

**役割**: 前提条件から結論を導く

**原理**:
- 一般的な原理から個別の結論を導く
- **前提から論理的に展開する**

**実践**:
- 基盤システムから個別のシステムへと論理的に展開
- 前提条件を基に、システムを段階的に構築

### 3. 末広がりな思想（演繹的拡張）

**役割**: 基盤から始まって、徐々に広がっていく

**原理**:
- 小さな基盤から大きなシステムへと段階的に拡張
- **帰納法的収束ではなく、演繹的末広がり**

**実践**:
- Phase 1: 基盤システム（最小限の基盤）
- Phase 2-3: 業務実行システム（基盤の上に構築）
- Phase 4: 顧客接点システム（業務システムの上に構築）
- Phase 5: 管理統合システム（全システムを統合）

## 🎯 五十音表の対照構造

### あ行（主観・伝え手）

```
あ  い  う  え  お
```

**特徴**:
- 母音のみで構成され、子音を持たない
- 日本語の基本的な母音を表し、他の行の基礎となる
- **主観・伝え手**の概念

### わ行（客観・受け手）

```
わ  ゐ  う  ゑ  を
```

**特徴**:
- 子音「w」と母音の組み合わせで構成される
- **客観・受け手**の概念

### 対照的な構図

| あ行（主観・伝え手） | わ行（客観・受け手） | 関係性 |
|-------------------|-------------------|--------|
| あ（主観） | わ（客観） | 立場：伝え手 ↔ 受け手 |
| い（実行する側） | ゐ（実行される側） | 意思決定：実行する ↔ 実行される |
| う（存在） | う（存在） | **共通事項** |
| え（選ぶ側） | ゑ（選ばれる側） | 実践知識：選ぶ ↔ 選ばれる |
| お（蓄える側） | を（蓄えられる側） | 経験知識：蓄える ↔ 蓄えられる |

## 📊 10の母音の定義

### 見る側の体系（主観・伝え手）

- **『あ』（主観）**: 個々の主観によって異なる。『う』というサービスを『あ』という主観で見る。
  - **ニーズ（需要）**: 主観として、サービスを受けたい！と考えるカスタマー
  - **シーズ（供給）**: 主観として、サービスを提供したい！と考えるデベロッパー
  - どちらも『あ』であり、それを受け取るのが『わ』

- **『い』（最終決定・実行する側）**: う→あ→お→えの後に続く『い』。これは最終決定だ
  - 実際に走らせるプログラムを意味する
  - 実行する側。ボタンを押した時に実行されるプログラム

- **『う』（有・共通事項）**: 有無の『有』。システム、WEBサイトが存在するかどうかということ。
  - システムの存在そのもの
  - システムは『う』を定義するために存在するもの
  - 『あ、い、う、え、お』と『わ、ゐ、う、ゑ、を』の両方の体系で共通

- **『え』（実践知識・選択・選ぶ側）**: 主観なので、どのような選択をするか？つまり見るものの選択
  - UIやUXはこの観点に入る
  - 実践知識（選択肢）として選ぶ側
  - 営業マンは案件を持ってくる側なので、主観

- **『お』（経験知識・蓄える側）**: 主観であるから、データをどのように活用しようかという経験に基づく知識
  - データの活用方法
  - 経験知識（経験則）として蓄える側

### 見られる側の体系（客観・受け手）

- **『わ』（客観）**: それを受け取る側（客観）
  - 『あ』（主観）が持つニーズやシーズを受け取る側
  - カスタマーのニーズを受け取る側
  - 開発者のシーズを受け取る側

- **『ゐ』（実行される側）**: 実行されたプログラム。つまり結果
  - 実行される側
  - 実行ボタン
  - 『ゐ』が実行ボタンならば、『い』はボタンを押した時に実行されるプログラム

- **『う』（有・共通事項）**: システムの存在そのもの（両方の体系で共通）

- **『ゑ』（選択された知識・選ばれる側）**: 選択された知識。つまり、選択がもたらす値を指す
  - 実践知識（選択肢）として選ばれる側
  - 清掃員は案件を見て対応するので客観

- **『を』（蓄積知識・蓄えられる側）**: 主観に対して客観であるから、単に蓄えられた知識でしかない
  - 経験知識（経験則）として蓄えられる側

## 🎯 システムの流れと対比関係

### システムの流れ: う→あ→お→え→い

1. **『う』（存在）**: システムの存在そのもの
2. **『あ』（主観）**: 個々の主観によって異なる。ニーズやシーズを持つ側
3. **『お』（経験知識）**: データを活用して分析・レポートを行う
4. **『え』（実践知識・選択）**: どのような選択をするか？見るものの選択
5. **『い』（最終決定）**: 実際に走らせるプログラム

### 対比関係: うわをゑゐ

1. **『う』**: 存在そのもの、共通事項（両方の体系で共通）
2. **『わ』**: それを受け取る側（客観）
3. **『を』**: 蓄えられる側（単に蓄えられた知識、データ）
4. **『ゑ』**: 選ばれる側（選択された知識）
5. **『ゐ』**: 実行される側（実行ボタン、実行された結果）

## 🎯 MDCSの階層構造：原理→原則→システム運用

### 原理を元に原則を作ってシステムを運用していく

**階層構造**:

1. **原理（Principle）**: 五十音表の対照構造
   - あ行（主観・伝え手） ↔ わ行（客観・受け手）
   - 日本語の根本構造に根ざした不変の基盤
   - 言語学的・文化的な基盤

2. **原則（Rule）**: 原理から導かれる一貫したルール
   - 伝え手と受け手の概念
   - ニーズとシーズの関係性
   - システムの流れ：う→あ→お→え→い
   - 対比関係：うわをゑゐ

3. **システム運用（Practice）**: 原則に基づいた実践
   - API設計
   - システム開発
   - コミュニケーション
   - データの送受信

### システムとプログラムの関係性

**前提**:
- **システムは原理であり、システムだけでは運用できない**
- **プログラムは原則であり、運用法則。法則だけではシステムは成り立たない**
- **システムとプログラムは同義であり、AはBが成立する**

**結論**:
- **したがって、返される値として、僕が作るシステムはC（システムとプログラムの統合）にならなければならない**

## 💡 実践的な応用

### APIにおける適用

- **データの記録**: あ行（記録する側） ↔ わ行（記録されるデータ）
- **データの返却**: あ行（返す側） ↔ わ行（返されるデータ）
- **送信と受信の違い**: 同じデータを扱っていても送信と受信で異なる場合がある
- **記録したと思っても、読み込んだ情報が違う**

### システム開発における適用

1. **要件定義**: あ行（ニーズ・シーズ） ↔ わ行（受け取る要件）
2. **設計**: あ行（設計者の意図） ↔ わ行（システムの実装）
3. **実装**: あ行（実行する側） ↔ わ行（実行される側）
4. **運用**: あ行（運用者の判断） ↔ わ行（システムの動作）

### コミュニケーションにおける適用

- **伝え手と受け手**: あ行（話し手、送信者） ↔ わ行（聞き手、受信者）
- **情報の伝達**: 同じ情報を扱っていても、送信と受信で異なる場合がある
- **人間同士のやり取りでも同じ構造**
