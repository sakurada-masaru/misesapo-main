# MDCSに則った開発環境構築ガイド

**作成日**: 2025年12月10日  
**目的**: 2人以上の開発者がGitHubでデータを共有し、同じCursorでシステムを構築するための環境改善

---

## 🎯 前提条件（三段論法による検証）

### 現在の前提条件

1. **GitHub**: サイトを公開している（前提条件）
2. **Cursor**: そのファイルを作る手段（開発環境）
3. **ファイル**: MDCSのシステム（このフォルダで実行されるシステム）
4. **開発者**: 2人以上の開発者が同じファイルを共有
5. **開発ツール**: 両者がCursorを使用

### 関係性の明確化

**構造**:
- **GitHub** = 公開・運用の基盤（前提条件）
- **Cursor** = 開発ツール（ファイルを作る手段）
- **ファイル** = MDCSのシステム（開発対象）

**開発フロー**:
```
Cursor（開発） → ファイル（MDCSのシステム） → GitHub（公開・運用）
```

**グループC**: 同じCursorツールを使用する（分類方法2: 開発環境）

**検証**:
- 開発者1が同じファイルを編集する（A1 ∈ B）

**結論**: 前提条件が正確であることを確認

## 💡 推奨される具体的な実践方法

### 1. ブランチ戦略

```
main（本番環境）
  └─ develop（開発環境）
      ├─ feature/機能A
      ├─ feature/機能B
      └─ feature/機能C
```

**ルール**:
- mainブランチは直接編集しない
- developブランチからfeatureブランチを作成
- featureブランチで作業後、developブランチにマージ
- developブランチで統合テスト後、mainブランチにマージ

### 2. ファイル分割戦略

**原則**:
- 大きなファイルを小さなファイルに分割
- 機能ごとにファイルを分ける
- 共通部分は別ファイルに分離

**例**:
- `src/pages/` - ページファイル
- `src/assets/css/` - CSSファイル
- `src/assets/js/` - JavaScriptファイル
- `src/data/` - データファイル

### 3. 作業領域の明確化

**原則**:
- 各開発者が担当する領域を明確にする
- 重複を避けるためのルールを確立
- 必要に応じて、領域を横断的に理解する

**例**:
- 開発者1: フロントエンド（HTML/CSS）
- 開発者2: バックエンド（Lambda/DynamoDB）
- 両者: 共通部分（MDCSの論理構造）

### 4. コミット・プッシュのルール

**コミットメッセージのルール**:
- `feat: 新機能の追加`
- `fix: バグの修正`
- `docs: ドキュメントの更新`
- `refactor: リファクタリング`
- `chore: その他の変更`

**プッシュのタイミング**:
- 作業が完了したら、すぐにプッシュ
- 競合を避けるため、定期的にプッシュ
- 大きな変更は、小さな単位に分けてプッシュ

### 5. レビューの標準化

**コードレビューのルール**:
- プルリクエストを作成
- レビューを依頼
- 承認後にマージ

**承認のルール**:
- 最低1人の承認が必要
- MDCSの論理に則っているか確認
- コードの品質を確認
