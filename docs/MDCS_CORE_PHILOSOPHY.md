# MDCS コア哲学（開発の主軸）

**作成日**: 2025年12月10日  
**目的**: システム開発と会社運営の核となる論理構造を定義

---

## 🎯 概要

**MDCS（ミセサポ演繹法分類体系）**は、システム開発と会社運営の核となる論理構造です。

**基本思想**: 「失敗しないシステムではなく、失敗しても良いシステム」

**MDCSの本質的な目的**:
- **失敗を恐れない開発環境の構築**: 厳格にするのは失敗しないためではなく、失敗を恐れないため
- **属人化の廃止**: 論理構造により、誰でも同じ判断ができる
- **相互補完的な開発環境**: 一人がみんなをカバーし、みんなが一人をカバーする状況を論理的に体現
- **層の厚い開発環境**: 複数の人が同じ領域を理解し、カバーできる環境を整える

**MDCSの本質**:
- **このシステムは一生完成することはない**: 常に可能性を秘めているから
- **完璧を目指すのではなく、常に改善し続ける**: 評価スコアが90でも問題ない、なぜならそれが正解だから
- **成長し続けるシステム**: 常に新しい可能性を秘め、進化し続ける

## 📐 論理構造（3段階）

### 1. 三段論法（前提の正確性の確保）

**役割**: 前提条件の正確性を確かめる

**MDCSでの三段論法の構造**:

**目的**: どの分類方法を使っても、同じ要素が同じ概念グループに属することを検証する

**論理構造**:
- **大前提**: 要素Aが分類方法1のグループBに属する（A ∈ B）
- **小前提**: 要素Aが分類方法2のグループCに属する（A ∈ C）
- **結論**: したがって、グループBとグループCは同じ概念グループを表している（B ≡ C）

**システムとプログラムの関係性（三段論法の実践例）**:

**前提**:
- **システムは原理であり、システムだけでは運用できない**
- **プログラムは原則であり、運用法則。法則だけではシステムは成り立たない**
- **システムとプログラムは同義であり、AはBが成立する**

**論理構造**:
- **A = システム（原理）**: システムは原理であり、システムだけでは運用できない
- **B = プログラム（原則）**: プログラムは原則であり、運用法則。法則だけではシステムは成り立たない
- **AはB**: システムとプログラムは同義であり、AはBが成立する

**結論**:
- **したがって、返される値として、僕が作るシステムはCにならなければならない**
- **C = システムとプログラムの統合**: 原理（システム）と原則（プログラム）を統合した、運用可能なシステム

**この三段論法の意味**:
- システム（原理）だけでは運用できない
- プログラム（原則）だけではシステムは成り立たない
- システムとプログラムは同義であり、統合されなければならない
- したがって、返される値として、システムはC（システムとプログラムの統合）にならなければならない

**実践例**:
- A = パブリック関連（要素）
- B = 顧客接点システム（分類方法1でのグループ）
- C = 顧客ドメイン（分類方法2でのグループ）
- **検証**: 
  - パブリック関連が顧客接点システムに属する（A ∈ B）
  - パブリック関連が顧客ドメインに属する（A ∈ C）
  - したがって、顧客接点システムと顧客ドメインは同じ概念グループを表している（B ≡ C）

**簡潔な表現**:
- 要素AがグループBに属し、要素AがグループCに属するなら、グループBとグループCは同じ概念グループである

**前提条件の検証**:
- 複数の要素（A1, A2, A3...）について、同じ検証を行う
- 全ての要素でB ≡ Cが成り立つことを確認
- これにより、分類方法間の一貫性が確保される

**実践**:
- どの分類方法を使っても、同じ要素が同じ概念グループに属することを検証
- 分類方法間の一貫性を確保
- 前提条件の正確性を保証

### 2. 演繹法（前提から結論への展開）

**役割**: 前提条件から結論を導く

**原理**:
- 一般的な原理から個別の結論を導く
- **前提から論理的に展開する**

**実践**:
- 基盤システムから個別のシステムへと論理的に展開
- 前提条件を基に、システムを段階的に構築

**演繹法の実践例**:
- **前提（一般的な原理）**: 「失敗しないシステムではなく、失敗しても良いシステム」
- **展開**: この基本思想から論理的に展開する
- **結論（個別の結論）**: 「このシステムは一生完成することはない。なぜなら常に可能性を秘めているから」
- **検証**: 評価スコアが90でも問題ない、なぜならそれが正解だから

**この実践例が示すこと**:
- MDCS自体が演繹法によって構築されている
- 基本思想から論理的に展開することで、新しい認識に到達する
- 演繹法により、MDCS自体も進化し続ける

### 3. 末広がりな思想（演繹的拡張）

**役割**: 基盤から始まって、徐々に広がっていく

**原理**:
- 小さな基盤から大きなシステムへと段階的に拡張
- **帰納法的収束ではなく、演繹的末広がり**

**実践**:
- Phase 1: 基盤システム（最小限の基盤）
- Phase 2-3: 業務実行システム（基盤の上に構築）
- Phase 4: 顧客接点システム（業務システムの上に構築）
- Phase 5: 管理統合システム（全システムを統合）

## 💭 論理立てることの重要性

### なぜ論理立てることが重要なのか

#### 1. 「前提を間違えると間違えてしまう」という認識

**問題**: 前提条件が間違っていると、その後の全ての判断が間違った方向に進む

**解決**: 三段論法により、前提条件の正確性を検証する
- 要素AがグループBに属するか？（A ∈ B）
- 要素AがグループCに属するか？（A ∈ C）
- グループBとグループCは同じ概念グループか？（B ≡ C）

**重要性**: 前提条件を正確に検証することで、その後の判断が正しい方向に進む

#### 2. 「失敗しても良いシステム」の実現

**問題**: 論理が一貫していないと、失敗の影響範囲が予測できない

**解決**: 論理構造を明確にすることで、失敗の影響範囲を限定できる
- 前提条件が明確 → 失敗の影響範囲が明確
- 論理的一貫性が確保 → 失敗からの回復が容易

**重要性**: 論理立てることで、「失敗しても良いシステム」を実現できる

#### 3. 関係者への伝達

**問題**: 論理が不明確だと、関係者間で理解が異なる

**解決**: 論理構造を明確にすることで、関係者間で同じ理解を共有できる
- 三段論法による検証 → 誰でも同じ結論に到達できる
- 明確な論理構造 → コミュニケーションが円滑になる

**重要性**: 論理立てることで、関係者間の理解を統一できる

#### 4. システムの破綻防止

**問題**: 論理が一貫していないと、システム全体が破綻する

**解決**: 論理構造を明確にすることで、システムの整合性を保証できる
- 分類方法間の一貫性 → システムの整合性
- 前提条件の正確性 → システムの信頼性

**重要性**: 論理立てることで、システムの破綻を防止できる
